延时从库 *****

1、介绍

是我们配置的一种特殊从库，人为配置从库和主库延时N小时

2、为什么要有延时从库

普通主从复制非常擅长物理损坏，但没有办法解决逻辑损坏

3、配置延时从库

SQL线程延时：数据已经写入relay-log中，SQL线程“慢点”运行
一般企业建议3-6小时，具体看公司运维人员对于故障的反应时间
mysql>stop slave;
mysql>CHANGE MASTER TO MASTER_DELAY = 300;
mysql>start slave;
mysql> show slave status \G
SQL_Delay: 300
SQL_Remaining_Delay: NULL

4、延时从库应用

4.1、故障恢复思路

1）一主一从，从库延时5分钟，主库误删除1个库
2）5分钟之内，侦测到误删除操作
3）停止从库SQL线程
4）截取relay-log日志
   起点：停止SQL线程时，relay-log最后应用位置
   show slave status \G
   Relay_Log_File: mysqld-relay-bin.000002
   Relay_Log_Pos: 320
   终点：误删除之前的positon（GTID）
   show relaylog events in 'db01-relay-bin.000002'	
5）恢复截取的日志到从库
情况一：就一个库的话，从库替代主库工作
情况二：从库导出故障库，还原到主库中

4.2、故障模拟及恢复

1）主库数据操作
mysql> create database relay charset utf8;
mysql> use relay
mysql> insert into t1 values(1);
mysql> drop database relay;

2）停止从库SQL线程
mysql> stop slave sql_thread;

3）截取relay-log的起点和终点
起点：
mysql> show slave status \G
Relay_Log_File: mysqld-relay-bin.000002
Relay_Log_Pos: 320
终点：
mysql> show relaylog events in 'mysqld-relay-bin.000002';

4）从库恢复relay-log
set sql_log_bin=0;
source /tmp/relay.sql
set sql_log_bin=1;

5）从库身份解除
stop slave;
reset slave all

过滤复制（有选择性的复制） ***

1、简介

不能与延时从库为同一节点，延时从库不对外提供服务，只是为了备份恢复时使用
IO线程截取日志时有 -d 选项、指定库截取日志，根据use语句记录不同库的日志
SQL线程基于某个库进行回放，这种比较合适一些
主库：
show master status;
Binlog_Do_DB                 #白名单
Binlog_Ignore_DB             #黑名单
二选一，Binlog_Do_DB=库名字，多行不能用逗号分隔，记录日志的库
从库：
show slave status\G
Replicate_Do_DB:             #库级别的，应用最多
Replicate_Ignore_DB: 
Replicate_Do_Table:          #表级别的
Replicate_Ignore_Table: 
Replicate_Wild_Do_Table:     #模糊
Replicate_Wild_Ignore_Table:

2、快速恢复测试环境

从库：
mysql> drop database relay;
主库：
mysql -S /data/3307/mysql.sock
mysql> reset master;
从库：
CHANGE MASTER TO 
MASTER_HOST='10.0.0.51',
MASTER_USER='repl',
MASTER_PASSWORD='123',
MASTER_PORT=3307,
MASTER_LOG_FILE='mysql-bin.000001',
MASTER_LOG_POS=154,
MASTER_CONNECT_RETRY=10;
start slave;

3、过滤复制例子

从库：

systemctl restart mysqld3308
主库：
mysql> create database word;

半同步复制 ***

1、介绍

解决主从复制数据一致性（安全问题）
从库relay-log写入磁盘，IO线程会返回给主库ACK_reciver线程一个ACK，主库事务才能提交，
如果一直ACK没收到，超过10秒钟会切换为异步复制
性能不太好，用的不多

2、配置

加载插件
主:
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
从:
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';
查看是否加载成功:
show plugins;
启动:
主:
SET GLOBAL rpl_semi_sync_master_enabled = 1;
从:
SET GLOBAL rpl_semi_sync_slave_enabled = 1;
重启从库上的IO线程
STOP SLAVE IO_THREAD;
START SLAVE IO_THREAD;
查看是否在运行
主:
show status like 'Rpl_semi_sync_master_status';
从:
show status like 'Rpl_semi_sync_slave_status';

GTID复制 *****

1、GTID介绍

GTID（Global Transaction ID）是一个对已提交事务的唯一编号，并且是一个全局（主从复制）唯一的编号
官方定义如下：
GTID = server-uuid ：transaction_id
server-uuid在数据库初始化后第一次启动时就会生成，不用乱删除
[root@mysqld data]# cat auto.cnf 
[auto]
server-uuid=923a6d2c-9dc4-11ec-922c-000c299152c6
TID：是一个自增长的数据，从1开始
923a6d2c-9dc4-11ec-922c-000c299152c6:1-15
核心特性：全局唯一，幂等性

2、GTID核心参数

gtid-mode=on                        --启用gtid类型，否则就是普通的复制架构
enforce-gtid-consistency=true       --强制GTID的一致性
log-slave-updates=1                 --slave更新是否记入日志

3、GTID复制配置

3.1、清理环境

pkill mysqld
 \rm -rf /data/mysql/data/*
 \rm -rf /data/binlog/*

3.2、准备配置文件

主库db01：
cat > /etc/my.cnf <<EOF
[mysqld]
basedir=/data/mysql/
datadir=/data/mysql/data
socket=/tmp/mysql.sock
server_id=51
port=3306
secure-file-priv=/tmp
autocommit=0
log_bin=/data/binlog/mysql-bin
binlog_format=row
gtid-mode=on
enforce-gtid-consistency=true
log-slave-updates=1
[mysql]
prompt=db01 [\\d]>
EOF

slave1(db02)：
cat > /etc/my.cnf <<EOF
[mysqld]
basedir=/data/mysql
datadir=/data/mysql/data
socket=/tmp/mysql.sock
server_id=52
port=3306
secure-file-priv=/tmp

binlog_format=row
gtid-mode=on

EOF

slave2(db03)：
cat > /etc/my.cnf <<EOF
[mysqld]
basedir=/data/mysql

server_id=53
port=3306

3.3、初始化数据

mysqld --initialize-insecure --user=mysql --basedir=/data/mysql  --datadir=/data/mysql/data

3.4、启动数据库

cat >/etc/systemd/system/mysqld.service <<EOF
[Unit]
Description=MySQL Server
Documentation=man:mysqld(8)
Documentation=http://dev.mysql.com/doc/refman/en/using-systemd.html
After=network.target
After=syslog.target

[Install]
WantedBy=multi-user.target

[Service]
User=mysql
Group=mysql
ExecStart=/application/mysql/bin/mysqld --defaults-file=/etc/my.cnf
LimitNOFILE = 5000
EOF
systemctl start mysqld

3.5、构建主从

master： 51
slave： 52，53

51：
grant replication slave  on *.* to repl@'10.0.0.%' identified by '123';

52\53：
change master to 
master_host='10.0.0.51',
master_user='repl',
master_password='123' ,
MASTER_AUTO_POSITION=1;

start slave;

3.6、GTID复制和普通复制的区别

CHANGE MASTER TO
MASTER_HOST='10.0.0.51',
MASTER_USER='repl',
MASTER_PASSWORD='123',
MASTER_PORT=3307,
MASTER_LOG_FILE='mysql-bin.000001',
MASTER_LOG_POS=444,
MASTER_CONNECT_RETRY=10;

change master to 
master_host='10.0.0.51',
master_user='repl',
master_password='123' ,
MASTER_AUTO_POSITION=1;
start slave;
1）在主从复制环境中，主库发生过的事务，在全局都由唯一GTID记录，更方便Failover
2）额外功能参数（3个）
3）change master to的时候不再需要文件名和position号，MASTER_AUTO_POSITION=1;
4）在复制过程中，从库不再依赖master.info文件名和position号，而是直接读取最后一个relay-log的GTID号
5）mysqldump备份时，默认会将备份中包含的事务操作，以以下方式
  SET @@GLOBAL.GTID_PURGED='8c49d7ec-7e78-11e8-9638-000c29ca725d:1';
  告诉从库，我的备份中已经有以上事务，你就不用运行了，直接从下一个GTID开始请求binlog就行
